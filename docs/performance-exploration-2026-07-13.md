# GXPlayer 性能探索评估（2026-07-13）

## 结论摘要

本次只做测量、代码路径审计和建议排序，没有为了追求数字而修改启动流程、音频内核或沙箱契约。

当前最值得优先处理的三项是：

1. 缓存启动恢复改为“正常 manifest 快速路径”，避免每次启动全量扫描 sidecar。
2. 收敛首屏后的无条件 IPC，把历史、完整缓存列表、设备枚举等延迟到对应页面。
3. 重新评估中文字体策略。字体占当前前端打包体积约 79.6%，是最确定的体积来源。

目前没有可复现的冷启动、暖启动毫秒基线，因此本文不声称具体时间收益。后续应先补阶段埋点，再对同一 release 产物重复测量。

## 启动关键路径审计

主窗口在 `src-tauri/tauri.conf.json` 中初始隐藏。`src-tauri/src/lib.rs::run` 在进入 `setup` 前同步完成 Windows identity 初始化和音频工作线程创建；`setup` 随后执行各存储、音源、沙箱和托盘初始化，并在闭包末段才恢复位置、显示主窗口。因此以下步骤都直接影响“进程启动到首次可见”：

- Windows identity 初始化与 `LocalAudioEngine::new`。音频引擎此时只建立通道和工作线程，不枚举或打开设备；设备选择发生在真正播放时。
- 打开曲库 SQLite，并检查/创建表和索引。
- 打开缓存目录，清理临时文件、读取 manifest、恢复 sidecar、写回 settings/manifest。
- 读取和迁移音源配置，扫描 drop-in 目录并逐个读取、校验、导入脚本。
- 创建隐藏的音源沙箱 WebView。
- 创建系统托盘，安装任务栏缩略图按钮。
- 最后才显示主窗口；SMTC/媒体会话在显示之后启动后台线程，不属于首次显示前的主要阻塞项。

### 最明确的随数据量增长热点：缓存恢复

`crates/gx-cache/src/lib.rs` 当前启动路径会：

- 遍历缓存目录清理临时文件；
- 读取 manifest；
- 即使 manifest 正常，也调用 sidecar 恢复扫描；
- 初始化结束时同步写回 settings 和 manifest。

因此正常启动 I/O 会随缓存文件数量增长。这个结论来自代码路径审计，尚未用大缓存样本测出毫秒值。

### 音源与沙箱

drop-in 扫描只检查一层目录，但每个 `.js` 都会读取、校验并进入导入/持久化路径。随后程序在主窗口显示前创建第二个隐藏 WebView，沙箱页面还要加载和解析约 427 KiB JS。

这不违反在线“不预解析”：它没有提前解析队列歌曲；但它仍属于启动初始化，可以考虑延后到窗口显示后，或在确实需要音源能力时启动。

### 首屏后的前端争用

React 首次 effect 在首屏渲染后并发触发：

- 音源列表与状态；
- 缓存状态、在线收藏和全部缓存条目；
- 最近 500 条历史；
- 音频设备枚举；
- 曲库、收藏、歌单及缺失文件扫描；
- 恢复队列引用的本地文件校验。

这些任务不一定延迟窗口出现，但会与首次交互竞争 CPU、磁盘和 IPC。尤其是完整缓存列表、500 条历史、全库缺失扫描、设备枚举，在用户没有进入对应页面时也会执行。

主界面尚未使用 `React.lazy` 或页面级动态导入，约 318 KiB 的主 JS 需要整体解析。在线榜单和诊断日志明细已经是按需加载，不在首屏无条件请求中。

## 建议清单（按投入产出比排序）

### 1. 为缓存恢复增加正常启动快速路径

建议：

- manifest 完整有效时，不再无条件扫描全部 sidecar；
- 仅在 manifest 缺失、损坏或用户主动修复时执行全量 sidecar 恢复；
- 临时文件清理和深度一致性检查移到窗口显示后的后台任务；
- settings/manifest 内容没有变化时不重复写盘。

预期：正常启动从随缓存条目数线性扫描，收敛为读取少量配置和单个 manifest。实际时间收益待测。

风险与验证：必须保留崩溃恢复能力。使用 0、100、10,000 条缓存，覆盖正常、缺 manifest、损坏 manifest、残留临时文件四组场景，比较恢复正确性、磁盘读取量和启动时间。

### 2. 将非首屏原生初始化移出首次显示关键路径

候选项：

- drop-in 重扫描和批量导入；
- 隐藏沙箱 WebView 创建与脚本加载；
- 可延后的任务栏 toolbar 初始化。

SourceStore 的基础配置仍可先加载；沙箱延后不能改变沙箱隔离，也不能引入队列预解析。UI 应在短暂延后期间明确显示“音源正在初始化”。

验证有/无音源、大量 drop-in、沙箱启动失败等场景，确认本地播放和首屏不依赖沙箱。

### 3. 收敛 React 首次 effect

建议首屏只请求当前页面、播放栏和恢复队列真正需要的数据：

- 完整缓存条目仅进入曲库缓存区时读取；
- 历史仅进入历史页时读取；
- 设备列表仅进入设置页或展开设备选择时枚举；
- 缺失文件扫描延后到空闲时段，或仅校验恢复队列引用的路径；
- 为缓存、曲库等提供轻量摘要命令，完整数据进入页面再取。

预期：降低首屏后的 CPU/磁盘争用，数据量越大收益越明显。代价是页面首次进入需要短暂中文 loading 状态。

### 4. 评估“系统中文字体 + 小型品牌字体”

字体占 `dist` 79.6%，体积收益最确定。优先实验：

- 中文正文使用 Windows 系统字体栈；
- 品牌、数字和等宽区域保留 Geist/Geist Mono；
- 或只保留明确需要的字重与字符范围。

不建议只按当前静态中文文案做子集：歌名、歌词、作者和用户导入音源的运行时字符不可预测。

理论上前端目录最多可减少接近 4.45 MiB；安装包实际缩减受压缩率影响，必须重新构建测量。验证应覆盖简繁中文、日文、韩文、拉丁、罕见歌手名和歌词混排。

### 5. 页面级代码分割

把设置、音源管理、诊断日志、文本导入等非首屏 UI 拆成懒加载 chunk，降低主 JS 的解析和编译成本。

沙箱继续保持独立入口；密码学和压缩兼容依赖只有在用户脚本回归测试充分时才能调整。

验证主 chunk 大小、首屏脚本执行时间、首次切页延迟，以及离线安装包是否包含全部 chunk。

### 6. 图片和 CSS 的低风险清理

- 对 239,677 B PNG 做无损/近无损重压缩，并检查透明边缘和高 DPI；
- 用构建分析确认主题规则是否重复、是否有未引用选择器；
- 字体策略改变后重新统计 CSS，因为大量 `@font-face` 本身也占 CSS 体积。

收益小于前四项，应后做。

### 7. Rust release 体积实验

当前 workspace 没有显式 release profile。建议先用二进制 section/依赖分析确认 SQLite bundled、TLS/HTTP、音频编解码、Tauri/Windows glue 的真实占比，再分别实验：

- `strip = "symbols"`；
- Thin LTO；
- 更少的 codegen units。

不要直接启用 `panic = "abort"`：项目包含需要隔离故障的 `catch_unwind` 边界，改变 panic 策略需要单独评估。

每个选项应单独构建，记录 EXE、NSIS、MSI、构建时间、冷/暖启动和完整测试结果。

## 后续测量方案

1. 固定同一个最终 release 产物，冷启动和暖启动各至少 20 次，报告中位数与 P95。
2. 增加阶段时间戳：进程入口、setup 各阶段、主窗口 `show`、React 首次 commit、`GX_PHASE0_UI_READY`、首批必要数据完成、首次可交互。
3. 场景至少覆盖：空数据、日常数据、大曲库、大缓存、大量 drop-in。
4. Rust 侧用 `Instant` 分段；Windows 侧用系统性能记录工具观察文件 I/O、线程和 WebView2；前端用 Performance API/WebView2 profiler。
5. 安装包体积只记录最终提交后重新构建的结果，不能沿用本文 18:20 的 release 快照。

## 本次决定

本提交只交付评估文档，不实施启动时序、字体或 release profile 改动。原因是三类改动都需要基线测量和专项回归；在缺少冷/暖启动数据时直接优化，无法可靠判断收益，也容易破坏缓存恢复、沙箱隔离或跨字符显示。
